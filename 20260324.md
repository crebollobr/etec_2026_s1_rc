## 📅 A Jornada do Pacote e o Roteamento

Nesta aula, vamos focar na estrada. Como os dados viajam de um ponto ao outro? A rede não é mágica, ela é feita de pequenos "pacotes" que saltam de roteador em roteador até o destino final.

### 📖 3. Simulação: Acessando o Google (O que acontece por trás dos panos)

Vamos imaginar você digitando `www.google.com` no seu navegador. Agora, vamos adicionar a camada de **Transporte (TCP)** à nossa jornada:

1.  **DNS (A Lista Telefônica):** Seu computador não entende nomes. Ele pergunta ao servidor DNS: *"Qual é o IP do Google?"*. O DNS responde: `142.250.190.46`.
    

2.  **A Decisão Local (Mascara):** Seu computador compara o IP do Google com o seu próprio IP e Máscara. Ele percebe: *"Esse IP não é meu vizinho, está fora da minha rede local"*.

3.  **A "Carta" com Endereço e Porta:** O computador prepara os dados e coloca neles as etiquetas de endereçamento. Para o Google, usamos o **HTTPS (Porta 443)**.
    * **Porta de Origem (Seu PC):** Uma **Porta Alta** aleatória (ex: `58234`). É por onde o seu navegador ficará "ouvindo" a resposta.
    * **Porta de Destino (Google):** **Porta 443** (Padrão mundial para tráfego web seguro/HTTPS).
    * **Endereço Lógico (IP):** Origem (Seu IP) / Destino (`142.250.190.46`).
    * **Endereço Físico (MAC):** Origem (Seu MAC) / Destino (**MAC do seu Gateway/Roteador**).
    

4.  **O Gateway (A Porta de Saída):** Como o IP de destino é externo, seu computador entrega o pacote para o Roteador. O roteador olha apenas para o IP de destino, ignora as portas por enquanto, e decide para qual vizinho na internet deve passar a "bola".

5.  **O Salto e o Roteamento:** O pacote salta de roteador em roteador. Cada roteador descarta o MAC antigo e coloca o MAC do próximo salto, mas o **IP e as Portas TCP permanecem os mesmos** até o destino final.

6.  **A Entrega e o Retorno:** O servidor do Google recebe o pacote na porta 443. Ele processa o pedido e gera uma resposta.
    * Agora, o Google inverte tudo: o destino da resposta será o seu IP e aquela **sua porta alta** (`58234`) que você deixou aberta esperando.
    * Esse pacote de retorno pode percorrer um caminho totalmente diferente pelos roteadores da internet!
    

---

### 💡 Por que a Porta Alta?
Se você abrir 10 abas do navegador ao mesmo tempo para o Google, seu computador usará uma **Porta Alta** diferente para cada aba (ex: 58234, 58235, 58236...). É assim que o seu sistema operacional sabe exatamente qual resposta do Google pertence a qual aba!


### 💻 Laboratório Prático 01 (1 hora)

Abram o terminal no nosso servidor Linux. Vamos rastrear essa jornada na prática!

**Passo 1: Quem são meus vizinhos?**
Vamos descobrir o nosso IP, quem é o nosso roteador (Gateway) e o endereço físico dele.
1. Veja o seu endereço IP e a rota padrão (Gateway):
   ```bash
   ip route
   ```
   *(Procure pela linha que começa com `default via`. Esse IP é o seu roteador!)*
2. Veja o endereço físico (MAC Address) do seu roteador:
   ```bash
   ip neigh

   arp -n
   ```
3. Veja as conexões ativas:
   ```bash
   ss -na
   ```
**Passo 2: A Corrida de Revezamento (Traceroute)**
Vamos ver por quantos roteadores nosso pacote passa até chegar ao Google.
```bash
traceroute google.com
```
*O que está acontecendo? O comando está listando cada roteador (cada "salto") no meio da internet que precisou tomar uma decisão de encaminhamento até chegar no destino.*



Excelente ideia! Esse laboratório "em dupla" é perfeito para materializar a teoria, pois separa quem **observa a rede** de quem **gera o tráfego**. Isso simula exatamente como um administrador de redes trabalha para debugar problemas.

## 💻 Laboratório Prático 02: A Dança dos Pacotes (TCP Handshake)

Neste lab, vamos ver o exato momento em que seu computador "pede licença" para o servidor e estabelece uma conexão confiável.

### 👥 Organização em Duplas:
* **Aluno A (O Observador):** Vai monitorar os cabos virtuais do servidor.
* **Aluno B (O Cliente):** Vai realizar o acesso ao site.

---

### 🔍 Passo 1: Descobrindo o Alvo (Dupla)
Escolham um site (ex: `pt.wikipedia.org`). Primeiro, precisamos do "CPF" (IP) desse site, pois a rede não entende nomes.

**Aluno A ou B executa:**
```bash
dig +short pt.wikipedia.org
```
*Anote o endereço IP que aparecer (ex: `103.102.166.224`). Vamos chamá-lo de `<IP_DO_SITE>`.*

---

### 🛡️ Passo 2: Preparando a Escuta (Aluno A)
O Aluno A vai usar o `tcpdump`, uma ferramenta poderosa que "grampeia" a placa de rede para ver o que está passando.

**Aluno A executa (Substitua o IP):**
```bash
sudo tcpdump -ni any host <IP_DO_SITE> and port 443 and tcp[tcpflags] & (tcp-syn|tcp-ack) != 0
```
* `-n`: Não resolve nomes (mostra IPs puros).
* `-i any`: Ouve em todas as interfaces de rede.
* `port 443`: Filtra apenas tráfego HTTPS (seguro).

---

### 🚀 Passo 3: O Ataque/Acesso (Aluno B)
Agora que o Aluno A está ouvindo, o Aluno B fará a requisição.

**Aluno B executa:**
```bash
curl -I https://pt.wikipedia.org/wiki/Página_principal
```
*(O parâmetro `-I` serve para pedir apenas o cabeçalho, para não inundar a tela de código HTML).*

---

### 📊 Passo 4: Analisando o Resultado (A hora da verdade!)

O **Aluno A** verá linhas surgindo no terminal. Procurem pelas "Flags" entre colchetes `[ ]`. É aqui que a mágica do TCP acontece:

1.  **O Pedido `[S]`: (SYN)** Seu computador dizendo "Quero sincronizar!".
2.  **A Resposta `[S.]`: (SYN-ACK)** O servidor da Wikipedia dizendo "Eu aceito! Vamos sincronizar!".
3.  **A Confirmação `[.]`: (ACK)** Seu computador dizendo "Ok, recebido! Vamos conversar".


#### 💡 O que observar no console:
* **Portas Dinâmicas:** Note que o seu computador usa uma porta alta e aleatória (ex: `45672`) para falar com a porta padrão `443` do servidor.
* **Independência:** Cada linha que você vê é um pacote independente que viajou pela rede.

---

### ❓ Desafio para a dupla:
Olhando a saída do `tcpdump`, quem mandou o primeiro pacote? O seu servidor ou a Wikipedia? Por que isso sempre acontece nessa ordem em um acesso web?

---

**Dica para o professor:** Se o `tcpdump` mostrar muita coisa, você pode pedir para os alunos adicionarem o filtro `tcp[tcpflags] & (tcp-syn|tcp-ack) != 0` para isolar apenas o handshake, mas a leitura crua (como passei acima) costuma ser mais rica para eles verem o volume de dados depois que a conexão estabiliza.
