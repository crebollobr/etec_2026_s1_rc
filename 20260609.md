## 📑 Parte 1: Base Teórica (40 min)

### 1. A Camada de Aplicação no TCP/IP

O modelo **TCP/IP** tem 4 camadas. Lembrando a tabela que já usamos:

| Camada TCP/IP | Função | Exemplos |
|---|---|---|
| **Aplicação** | Fala a língua do usuário e dos programas | **HTTP, HTTPS, DNS, SMTP** |
| Transporte | Entrega fim-a-fim, portas | TCP, UDP |
| Internet | Endereço lógico e rota | IP, ICMP |
| Acesso à Rede | Bits no fio/rádio | Ethernet, Wi-Fi |

A camada de aplicação é a **única que o programador toca diretamente** no dia a dia. Quando você abre um site, é o HTTP (camada de aplicação) que decide *o que* pedir; as camadas de baixo só cuidam de *como* os bytes chegam lá. O HTTP **não se preocupa** se a rede é cabo ou Wi-Fi, se passou por 2 ou 20 roteadores — ele entrega isso pro TCP e pronto. Isso se chama **abstração em camadas**, e é a ideia mais importante de redes.

> 💡 **Sacada:** a camada de aplicação trata de **mensagens** (um pedido, uma resposta). A de transporte trata de **portas e conexões**. A de internet, de **pacotes e IPs**. Cada uma resolve um problema e confia na de baixo.

### 2. HTTP é texto, e isso é de propósito

O HTTP foi desenhado para ser **legível por humanos**. Um pedido é só texto seguindo um formato:

```
MÉTODO  /caminho  HTTP/1.1      ← linha de pedido
Cabeçalho: valor                ← um ou mais cabeçalhos (headers)
Cabeçalho: valor
                                ← LINHA EM BRANCO (separa cabeçalho do corpo)
corpo (opcional)                ← os dados, quando existem
```

### 3. Os métodos (verbos) — o coração da aula

O **método** diz **qual ação** você quer fazer com o recurso. Pense numa planilha de recados:

| Método | Intenção | Na planilha de recados | Tem corpo? |
|---|---|---|---|
| **GET** | Ler | "me mostra os recados" | não |
| **POST** | Criar | "adiciona este recado novo" | sim |
| **PUT** | Substituir | "troca o recado nº 3 por este" | sim |
| **DELETE** | Apagar | "apaga o recado nº 3" | não |
| **HEAD** | Ler só os cabeçalhos | "existe? qual o tamanho?" | não |
| **PATCH** | Alterar um pedaço | "muda só uma palavra do recado 3" | sim |
| **OPTIONS** | Perguntar o que pode | "quais métodos você aceita?" | não |

Dois conceitos que caem em prova e em entrevista:

- **Seguro (safe):** não muda nada no servidor. `GET` e `HEAD` são seguros — você pode repetir à vontade.
- **Idempotente:** repetir dá o mesmo resultado final. `PUT` e `DELETE` são idempotentes (apagar o recado 3 dez vezes = recado 3 apagado). `POST` **não** é — postar dez vezes cria dez recados!

> ⚠️ **Mito comum:** "GET vai na URL, POST vai escondido". Mentira. POST também trafega **em texto puro** no HTTP — você provou isso vendo senha no `tcpdump`/Wireshark. O que protege é o **HTTPS**, assunto da próxima aula.

### 4. Servir arquivo × processar lógica

Existem dois tipos de "servidor HTTP":

- **Servidor de arquivos** (ex.: **Nginx**, Apache): pega um arquivo do disco e devolve. Ótimo para HTML, imagem, CSS. Ele basicamente só entende **GET/HEAD**; se você mandar `DELETE`, ele responde `405 Method Not Allowed`.
- **Servidor de aplicação** (ex.: seu script Python, Node.js): **roda código** para cada pedido e decide a resposta. É ele que implementa POST/PUT/DELETE com lógica.

Hoje você vai construir os **dois** e ver a diferença na prática.

### Etapa 2 — Seu servidor de aplicação (entende TODOS os verbos)

Crie o arquivo do **Mural de Recados** — um servidor que guarda recados na memória e responde a cada método com lógica própria:

```bash
cat > mural.py << 'EOF'
from http.server import BaseHTTPRequestHandler, HTTPServer
import json

recados = {}          # id -> texto
proximo_id = [1]      # contador (lista p/ poder alterar dentro dos métodos)

class Mural(BaseHTTPRequestHandler):

    def responder(self, codigo, dados):
        corpo = json.dumps(dados, ensure_ascii=False).encode('utf-8')
        self.send_response(codigo)
        self.send_header('Content-Type', 'application/json; charset=utf-8')
        self.send_header('Content-Length', str(len(corpo)))
        self.end_headers()
        self.wfile.write(corpo)

    def ler_corpo(self):
        tam = int(self.headers.get('Content-Length', 0))
        if tam == 0:
            return {}
        try:
            return json.loads(self.rfile.read(tam).decode('utf-8'))
        except Exception:
            return {}

    def do_GET(self):                      # LER todos os recados
        self.responder(200, {'recados': recados})

    def do_POST(self):                     # CRIAR um recado
        dados = self.ler_corpo()
        i = str(proximo_id[0]); proximo_id[0] += 1
        recados[i] = dados.get('texto', '')
        self.responder(201, {'criado': i, 'texto': recados[i]})

    def do_PUT(self):                      # SUBSTITUIR o recado /id
        ident = self.path.strip('/')
        if ident not in recados:
            return self.responder(404, {'erro': 'recado nao existe'})
        recados[ident] = self.ler_corpo().get('texto', '')
        self.responder(200, {'atualizado': ident, 'texto': recados[ident]})

    def do_DELETE(self):                   # APAGAR o recado /id
        ident = self.path.strip('/')
        if ident not in recados:
            return self.responder(404, {'erro': 'recado nao existe'})
        del recados[ident]
        self.responder(200, {'apagado': ident})

print('Mural ouvindo na porta 8000...')
HTTPServer(('0.0.0.0', 8000), Mural).serve_forever()
EOF
python3 mural.py
```

Deixe rodando neste terminal.

### Etapa 3 — Converse com seu servidor usando os 4 verbos (terminal 2)

Abra a 2ª aba do Killercoda (`+`) e mande um comando de cada vez, **lendo a resposta**:

```bash
# 1) GET — está vazio?
curl http://localhost:8000/

# 2) POST — criar dois recados (repare: cada POST cria um NOVO)
curl -X POST -d '{"texto":"Comprar cabo de rede"}' http://localhost:8000/
curl -X POST -d '{"texto":"Estudar para a prova"}' http://localhost:8000/

# 3) GET — agora veja os dois
curl http://localhost:8000/

# 4) PUT — substituir o recado 1
curl -X PUT -d '{"texto":"Comprar cabo CAT6"}' http://localhost:8000/1

# 5) DELETE — apagar o recado 2
curl -X DELETE http://localhost:8000/2

# 6) GET — confira o resultado final
curl http://localhost:8000/

# 7) DELETE num id que não existe — qual status?
curl -i -X DELETE http://localhost:8000/99
```

📝 **Anote para o relatório:**
- a) O que mudou no `GET` final depois do PUT e do DELETE?
- b) Rode o **POST** da etapa 2 mais 3 vezes seguidas. Quantos recados aparecem? Isso mostra que POST **não é idempotente**. Explique com suas palavras.
- c) Qual código o servidor devolveu no DELETE de um id inexistente (item 7)? De que **família** é?

### Etapa 4 — Espie o protocolo cru

Veja o que **realmente** trafega, com o `-v` (verbose) mostrando linha de pedido, cabeçalhos e corpo:

```bash
curl -v -X POST -d '{"texto":"teste cru"}' http://localhost:8000/ 2>&1 | head -25
```

📝 **Anote:** identifique na saída a **linha de pedido** (método + caminho + versão), pelo menos **2 cabeçalhos** que o curl enviou e a **linha de status** da resposta.

### 🏅 Desafio extra (vale ponto)

Implemente o método **`HEAD`** no `mural.py` (adicione `def do_HEAD(self):`). Ele deve responder igual ao GET, **mas sem o corpo** — só os cabeçalhos, incluindo um `Content-Length` correto. Teste com `curl -I http://localhost:8000/` e prove que vem cabeçalho mas não vem corpo.
