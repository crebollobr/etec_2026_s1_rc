## 📑 Parte 1: Base Teórica (45 min)

### 1. Dois jeitos de criptografar

- **Criptografia simétrica:** uma **única chave** tranca e destranca. Rápida, mas tem um problema: como combinar a chave secreta com alguém que você nunca encontrou, por uma rede que tem espiões?
- **Criptografia assimétrica:** um **par** de chaves matematicamente ligadas:
  - 🔓 **Chave pública** — você distribui à vontade. O que ela **tranca**, só a privada destranca.
  - 🔑 **Chave privada** — fica **só com você**, em segredo absoluto.

> 🧠 **Analogia do cadeado:** a chave pública é um **cadeado aberto** que você espalha pelo mundo. Qualquer um pode fechar uma caixa com seu cadeado e te mandar — mas só **você**, com a chave privada, abre. Ninguém precisou te entregar segredo nenhum antes.

O HTTPS usa **as duas**: a assimétrica no início (para combinar com segurança uma chave simétrica de sessão) e a simétrica depois (porque é mais rápida para o resto da conversa). O melhor dos dois mundos.

### 2. O que é, afinal, um certificado?

Ter uma chave pública não basta. Como sei que **aquela** chave pública é mesmo do `banco.com.br` e não de um impostor?

Um **certificado digital** resolve isso. Ele é um documento que contém, basicamente:

```
- A identidade:        CN = banco.com.br
- A chave pública do dono
- Validade:            de 01/01/2026 até 01/01/2027
- Quem garante isso:   assinatura digital de uma Autoridade Certificadora (CA)
```

Ou seja: **certificado = chave pública + identidade + assinatura de quem garante**. É a "carteira de identidade" do site. E, como toda identidade, ela só vale porque um **órgão confiável a assinou**.

### 3. A cadeia de confiança 🔗

Ninguém confia num certificado "do nada" — confia em **quem o assinou**. E isso forma uma corrente:

```
  CA Raiz (Root)         ← confiada DE FÁBRICA pelo navegador/sistema
      │ assina
  CA Intermediária       ← assinada pela raiz
      │ assina
  Certificado do site    ← o que o banco.com.br apresenta a você
```

Seu navegador já **nasce** com uma lista de CAs raiz em quem confia (a *trust store* — centenas delas: DigiCert, Let's Encrypt, Google Trust Services...). Quando você acessa um site:

1. O site manda o certificado dele + os intermediários;
2. O navegador segue a corrente assinatura por assinatura até chegar numa **raiz que ele conhece**;
3. Chegou numa raiz confiável, validade ok, domínio bate → 🔒 **cadeado**;
4. Quebrou em qualquer elo → ⚠️ **tela vermelha**.

> 💡 É por isso que **autoassinado dá aviso**: um certificado que assina a si mesmo não encosta em nenhuma raiz da lista do navegador. A criptografia funciona, mas a **identidade** não foi garantida por ninguém de fora.

### 4. As três perguntas que todo certificado precisa passar

Quando dá tela vermelha, é sempre uma destas três falhando:

| Pergunta | Se falhar... |
|---|---|
| A corrente chega numa **CA confiável**? | "autoridade desconhecida" (autoassinado cai aqui) |
| Está **dentro da validade**? | "certificado expirado" |
| O **domínio bate** com o do certificado? | "certificado de outro site" |

Hoje você vai fabricar cada uma dessas situações de propósito.

### Etapa 1 — Veja uma chave pública e uma privada com seus olhos

```bash
# gera o par de chaves (privada) ...
openssl genrsa -out minha_privada.pem 2048
# ... e extrai a parte PÚBLICA dele
openssl rsa -in minha_privada.pem -pubout -out minha_publica.pem

ls -l minha_*.pem
head -3 minha_privada.pem    # diz PRIVATE KEY
head -3 minha_publica.pem    # diz PUBLIC KEY
```

📝 **Anote:** qual delas você poderia publicar no seu site sem problema, e qual **nunca** pode vazar? Por quê?

**Prove a mágica do cadeado** — tranque com a pública, destranque com a privada:

```bash
echo "senha-secreta-do-banco" > segredo.txt
# qualquer um com a chave PÚBLICA tranca:
openssl pkeyutl -encrypt -pubin -inkey minha_publica.pem -in segredo.txt -out segredo.enc
cat segredo.enc            # lixo binário ilegível
# só a chave PRIVADA destranca:
openssl pkeyutl -decrypt -inkey minha_privada.pem -in segredo.enc
```

📝 **Anote:** o que apareceu no `cat segredo.enc` e o que apareceu na última linha? Tente destrancar com a **pública** (`-pubin`) e veja que **não dá** — registre o erro.

### Etapa 2 — Vire uma Autoridade Certificadora (crie sua CA raiz)

```bash
# chave privada da SUA CA (o segredo mais valioso de uma CA de verdade)
openssl genrsa -out ca_raiz.key 2048
# certificado raiz autoassinado da sua CA (validade longa, como as reais)
openssl req -x509 -new -nodes -key ca_raiz.key -sha256 -days 3650 \
  -out ca_raiz.crt -subj "/CN=ETEC Autoridade Raiz/O=Curso de Redes"

openssl x509 -in ca_raiz.crt -noout -subject -issuer
```

📝 **Anote:** repare que `subject` (quem é) e `issuer` (quem assinou) são **iguais**. Isso é a assinatura de uma **raiz** — ela assina a si mesma. Explique por que só a raiz pode fazer isso.

### Etapa 3 — Emita um certificado para o seu servidor, assinado pela sua CA

```bash
# 1) chave do servidor
openssl genrsa -out servidor.key 2048

# 2) pedido de certificado (CSR): "CA, por favor assine esta chave para o nome localhost"
openssl req -new -key servidor.key -out servidor.csr -subj "/CN=localhost"

# 3) a CA assina o pedido -> nasce o certificado do servidor
openssl x509 -req -in servidor.csr -CA ca_raiz.crt -CAkey ca_raiz.key \
  -CAcreateserial -out servidor.crt -days 365 -sha256

# confira a corrente: subject = localhost, issuer = sua CA
openssl x509 -in servidor.crt -noout -subject -issuer
```

📝 **Anote:** agora `subject` (localhost) e `issuer` (ETEC Autoridade Raiz) são **diferentes**. Você acabou de criar uma **cadeia de confiança** de 2 elos: CA raiz → servidor.

### Etapa 4 — Coloque o mural no ar com HTTPS

Copie seu `mural.py` da aula passada para cá (ou recrie) e crie uma versão HTTPS:

```bash
cat > /root/pki/mural_https.py << 'EOF'
from http.server import BaseHTTPRequestHandler, HTTPServer
import ssl, json

class Mural(BaseHTTPRequestHandler):
    def do_GET(self):
        corpo = json.dumps({'mensagem': 'Mural seguro via HTTPS!'}).encode()
        self.send_response(200)
        self.send_header('Content-Type', 'application/json; charset=utf-8')
        self.end_headers()
        self.wfile.write(corpo)

ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
ctx.load_cert_chain('/root/pki/servidor.crt', '/root/pki/servidor.key')
srv = HTTPServer(('0.0.0.0', 8443), Mural)
srv.socket = ctx.wrap_socket(srv.socket, server_side=True)
print('Mural HTTPS na porta 8443')
srv.serve_forever()
EOF
python3 /root/pki/mural_https.py
```

No **terminal 2**, teste a tela vermelha e a tela verde:

```bash
# 1) sem dizer em quem confiar -> ERRO (curl não conhece sua CA)
curl https://localhost:8443/

# 2) dizendo "confie na minha CA raiz" -> FUNCIONA, corrente validada
curl --cacert /root/pki/ca_raiz.crt https://localhost:8443/

# 3) ignorando a verificação (jeito perigoso) -> funciona, mas inseguro
curl -k https://localhost:8443/
```

📝 **Anote:** a mensagem de erro do item 1 e por que o item 2 passou. Em uma frase: contra **quem** o item 3 (`-k`) deixa você desprotegido?

> 💻 **No Windows com Nginx:** em vez do Python, aponte `ssl_certificate servidor.crt;` e `ssl_certificate_key servidor.key;` no `nginx.conf` (porta `8443 ssl`), como vocês fizeram em 02/06. Para o navegador parar de avisar, você **importa a `ca_raiz.crt` na loja de Autoridades Raiz Confiáveis do Windows** — e aí vê o cadeado ficar verde para um certificado que **você mesmo** assinou. É exatamente isso que uma empresa faz na rede interna dela.

### Etapa 5 — Inspecione certificados do mundo real

```bash
# a corrente completa do Google: leaf -> intermediária -> raiz
openssl s_client -connect www.google.com:443 -showcerts </dev/null 2>/dev/null \
  | grep -E "s:|i:"
```

📝 **Anote:** quantos elos a corrente tem? Quem é a CA raiz no topo?

Agora o **museu dos certificados quebrados** (badssl.com) — uma falha de cada tipo da teoria:

```bash
curl https://expired.badssl.com/        # validade vencida
curl https://wrong.host.badssl.com/     # domínio não bate
curl https://self-signed.badssl.com/    # autoassinado (sem CA confiável)
```

📝 **Anote:** o erro de cada um e a **qual das três perguntas** da teoria cada falha corresponde.
