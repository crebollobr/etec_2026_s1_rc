

# 🌐 Protocolo HTTP — Criptografia, HTTPS e Proxy Reverso com Nginx e Node.js

Hoje vamos compreender o funcionamento do protocolo HTTP, focando em dois pilares essenciais para a infraestrutura web: **a segurança de tráfego (HTTPS/TLS)** e a entrega de aplicações estruturadas em camadas utilizando o **Nginx como Proxy Reverso** para o **Node.js**.

---

## 📑 Parte 1: Base Teórica Essencial

Todo o conteúdo teórico exposto a seguir serve de fundamentação exclusiva para a execução das atividades práticas. Nada será cobrado sem antes ser explicado aqui.

### 1. Clientes, Servidores e a Infraestrutura Web

* **Clientes (User-Agent):** Aplicações que iniciam as requisições web, como navegadores (Chrome, Firefox), ferramentas de linha de comando (`curl`) ou sistemas automatizados.
* **Servidores HTTP:** Softwares que escutam portas de rede específicas para processar requisições e devolver recursos (páginas HTML, imagens, dados). Exemplos: Nginx e Apache.
* **Infraestrutura On-Premise vs Cloud:** * **On-Premise:** Servidores alocados fisicamente dentro da própria empresa, exigindo gerenciamento direto de hardware, energia e climatização.
* **Cloud (Nuvem):** Recursos computacionais virtualizados e contratados sob demanda em datacenters globais (AWS, Google Cloud, Azure), oferecendo alta escalabilidade.



### 2. Anatomia do Fluxo HTTP

O protocolo funciona em um ciclo de **Requisição (Request)** e **Resposta (Response)** baseado em texto plano.

* **URI e Métodos:** A URI determina o caminho do recurso (ex: `/index.html`). O Método (ou Verbo) indica a ação desejada: `GET` (leitura), `POST` (criação), `PUT`/`PATCH` (alteração) e `DELETE` (remoção).
* **Cabeçalhos (Headers) e Corpo (Body):** Os cabeçalhos transportam metadados da comunicação (como `Content-Type`, que define o formato do arquivo). O corpo contém os dados úteis da transmissão (código HTML, arquivos ou strings JSON).
* **Códigos de Status:** Respostas numéricas padronizadas enviadas pelo servidor:
* **2xx (Sucesso):** A requisição foi processada com êxito (Ex: `200 OK`).
* **3xx (Redirecionamento):** O recurso mudou de localização.
* **4xx (Erro do Cliente):** Falha na requisição enviada (Ex: `404 Not Found` para caminhos incorretos ou `401 Unauthorized` para falta de credenciais).
* **5xx (Erro do Servidor):** Falha interna no processamento da aplicação (Ex: `500 Internal Server Error`).


* **Sessões e Cookies:** Como o HTTP é um protocolo sem memória (*stateless*), os **Cookies** são pequenos arquivos de texto enviados pelo servidor e guardados pelo navegador para persistir estados, como sessões de usuários autenticados.

### 3. Segurança com HTTPS, TLS e Chaves Autoassinadas

Por padrão, o HTTP trafega em texto puro (porta `80`), permitindo a interceptação de dados na rede. O **HTTPS** (porta padrão `443`) adiciona uma camada de segurança utilizando o protocolo **TLS** (Transport Layer Security) para criptografar os dados antes do envio.

* **Criptografia Assimétrica:** Utiliza um par de chaves complementares. A **Chave Pública** é disponibilizada para que os clientes criptografem os dados enviados. A **Chave Privada** permanece sob controle estrito do servidor para realizar a descriptografia.
* **Chaves Autoassinadas:** Em ambientes de desenvolvimento, laboratórios ou redes internas corporativas, os analistas geram e assinam o próprio certificado digital (via ferramenta `openssl`). O tráfego de dados torna-se perfeitamente criptografado, embora o navegador exiba um aviso de atenção pelo fato de o certificado não ter sido emitido por uma Autoridade Certificadora (CA) comercial.
* **Restrições de Portas no Sistema Operacional:** Portas de rede de `0` a `1023` são consideradas privilegiadas e sua execução exige privilégios de Administrador do sistema. Para usuários com contas padronizadas de segurança (sem privilégios administrativos), utilizam-se **Portas Altas (acima de 1024)**, como a porta **`8443`** para tráfego HTTPS, dispensando permissões especiais de sistema.

#### Modelo de Configuração de HTTPS no Nginx (`nginx.conf`):

Para ativar o HTTPS em portas altas usando chaves locais no Nginx, a estrutura básica do bloco `server` requer o parâmetro `ssl` na porta e os caminhos dos arquivos gerados:

```nginx
server {
    listen       PORTA_ALTA ssl;
    server_name  localhost;

    ssl_certificate      NOME_DO_CERTIFICADO.crt;
    ssl_certificate_key  NOME_DA_CHAVE.key;
    
    # ... configurações de rotas
}

```

### 4. Ciclo de Execução do Node.js e Arquitetura de Proxy Reverso

O **Node.js** é um ambiente de execução que permite processar Javascript no lado do servidor. O ciclo padrão de implantação de um script Node.js compreende:

1. Instalação do runtime do Node.js no servidor de destino.
2. Inicialização do interpretador via terminal chamando o executável e o arquivo (`node arquivo.js`), o que ativa um processo contínuo escutando uma porta de rede interna (ex: porta `3000`).

#### O que é e como configurar um Proxy Reverso?

Em ambientes profissionais, não expomos o servidor de aplicação Node.js diretamente para a internet. Implementamos o **Nginx como Proxy Reverso**: ele atua como uma barreira frontal que recebe as requisições externas seguras (HTTPS na porta `8443`), gerencia a criptografia e redireciona o tráfego limpo internamente para a porta local onde o Node.js está ouvindo (`3000`).

Para fazer o Nginx encaminhar o fluxo de dados para outro serviço interno, limpamos as diretivas de arquivos locais (`root`/`index`) de dentro do bloco `location /` e aplicamos a seguinte sintaxe:

```nginx
location / {
    proxy_pass http://127.0.0.1:PORTA_INTERNA_DO_NODE;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

```

* `proxy_pass`: Define o destino interno (IP local `127.0.0.1` + porta da aplicação).
* `proxy_set_header`: Garante que os cabeçalhos com o IP real e o Host do cliente original não se percam no meio do caminho.

### ⚠️ Comportamento Crítico do Nginx no Windows (Aviso de Terminal)

Ao executar o comando `nginx.exe` ou `start nginx.exe` no terminal do Windows, **o prompt não ficará travado mostrando mensagens e retornará imediatamente para a linha de comando**. Isso é o comportamento normal do Nginx rodando em segundo plano (background).

**Atenção:** Se você digitar o comando `nginx.exe` várias vezes porque achou que não funcionou, você criará múltiplos processos fantasmas travando a mesma porta de rede. Se precisar aplicar alterações do arquivo de configuração, use o comando de recarga: `nginx.exe -s reload`. Se precisar derrubar processos travados, use `taskkill /f /im nginx.exe`.

## 🛠️ Atividade 1: Configuração do Nginx para Windows com HTTPS em Porta Alta

**Objetivo:** Configurar o servidor web Nginx em ambiente Windows utilizando uma conta de usuário padrão. Você gerará chaves criptográficas autoassinadas e definirá o Nginx para responder na porta alta **`8443`**.

### 📥 Passo 1: Preparação dos Arquivos

1. Baixe o pacote comprimido do Nginx para Windows (arquivo `.zip` oficial em `nginx.org`).
2. Extraia os arquivos em um diretório com permissão de escrita do seu usuário.

### 🔑 Passo 2: Geração do Certificado TLS Autoassinado

Abra o terminal disponível no sistema (como o Git Bash) dentro da pasta raiz do Nginx extraído e execute o comando abaixo para gerar a chave privada (`server.key`) e o certificado público (`server.crt`):

```bash
openssl req -x509 -nodes -days 365 -newkey rsa:2048 -keyout conf/server.key -out conf/server.crt

```

*(Você pode pressionar Enter para avançar por todas as perguntas organizacionais do terminal).*

### ⚙️ Passo 3: Configuração do Arquivo `nginx.conf`

Abra o arquivo `conf/nginx.conf` em um editor de texto e modifique o bloco `server` principal. Baseando-se no **Modelo de Configuração de HTTPS** visto na Seção 3 da teoria:

1. Altere a diretiva de escuta para a porta alta correta de segurança com suporte a SSL.
2. Insira as duas diretivas que apontam para os arquivos de certificado e chave gerados no passo anterior.
3. Certifique-se de manter a rota interna `location /` original apontando para a pasta padrão de arquivos locais `html`.

### 🚀 Passo 4: Inicialização e Validação

1. No Prompt de Comando (`cmd`), navegue até a pasta do Nginx e inicie o serviço apenas **uma vez**:
```cmd
nginx.exe

```


2. Abra o navegador e acesse o endereço completo especificando o protocolo seguro HTTPS e a porta alta configurada.
3. **Validação:** Confirme se o navegador exibe a mensagem de atenção esperada para chaves autoassinadas explicada na teoria e avance para checar a exibição da página padrão do Nginx.

## 🎯 Desafio Integrado: Implantação de Aplicação Node.js com Proxy Reverso Nginx

**Cenário:** O setor de desenvolvimento entregou uma aplicação em Node.js que opera na porta interna `3000`. Sua missão é colocar a aplicação em execução e reconfigurar o Nginx criado na atividade anterior para atuar como um **Proxy Reverso**, permitindo que o usuário externo acesse a aplicação com segurança através da porta HTTPS `8443`.

### 📋 Código fornecido da Aplicação (`app.js`):

Salve o script abaixo em uma pasta de sua escolha:

```javascript
const http = require('http');

const server = http.createServer((req, res) => {
  if (req.url === '/api/status') {
    res.writeHead(200, {'Content-Type': 'application/json; charset=utf-8'});
    res.end(JSON.stringify({ aplicativo: "Node.js", status: "Operacional" }));
  } else {
    res.writeHead(200, {'Content-Type': 'text/html; charset=utf-8'});
    res.end('<h1>🚀 Servidor Node.js respondendo via Proxy Reverso Nginx!</h1>');
  }
});

// Escuta a porta interna de aplicação
server.listen(3000, () => {
  console.log('Aplicacao Node.js iniciada na porta interna 3000.');
});

```

### 🛠️ Tarefas do Aluno:

1. **Executar a Aplicação Node.js:** Abra o terminal na pasta do arquivo `app.js` e utilize o interpretador nativo do Node.js aprendido na Seção 4 da teoria para iniciar o processo. O terminal deve permanecer ativo e exibindo o log de inicialização.
2. **Configurar o Nginx como Proxy Reverso:** Edite novamente o arquivo `conf/nginx.conf`. Dentro do bloco `location /`, remova as linhas que realizam a leitura de arquivos locais (`root` e `index`) e implemente o redirecionamento de tráfego.
* *Requisito:* Aplique a sintaxe exata da diretiva de encaminhamento HTTP abordada na seção de **Proxy Reverso** da teoria, apontando para o endereço local na porta utilizada pelo Node.js.
* *Requisito:* Adicione os dois cabeçalhos de preservação de Host e IP explicados na teoria.


3. **Recarregar e Testar o Ecossistema:**
* Lembre-se do aviso teórico sobre processos: não execute `nginx.exe` de novo. Use o comando correto de atualização de processos para aplicar as alterações do arquivo de configuração.
* Acesse pelo navegador a URL externa segura utilizando a porta alta do HTTPS. O navegador deve exibir de forma transparente a mensagem gerada pelo Node.js.
* Acesse a rota `/api/status` e certifique-se de que a resposta de dados estruturados JSON seja renderizada com sucesso.
