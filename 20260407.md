
# 🌐 Laboratório: Roteamento e NAT no Linux com QEMU



Nesta atividade prática, vamos simular uma rede corporativa utilizando o QEMU. Vamos entender na prática por que máquinas em redes diferentes não se comunicam por padrão e como transformar um servidor Linux em um roteador funcional com acesso à internet (NAT).

## 🗺️ O Cenário

Temos três máquinas virtuais emuladas, divididas em duas sub-redes isoladas e um roteador central:

* **Micro01:** `192.168.0.10` (Rede A)
* **Micro02 (Roteador):** `192.168.0.1` (Perna na Rede A), `192.168.1.1` (Perna na Rede B) e uma interface ligada à Internet.
* **Micro03:** `192.168.1.10` (Rede B)

---

## 🚀 Passo a Passo da Atividade

### 1. Preparação do Ambiente
1. Faça o download do arquivo `qemu.zip`.
2. Extraia o conteúdo na sua pasta atual (ex: `Downloads\qemu`).
3. Abra **3 janelas** do Prompt de Comando (CMD) e navegue até a pasta extraída em todas elas:
   ```cmd
   cd Downloads\qemu
   ```

### 2. Inicialização Estratégica
A ordem de inicialização importa. Como o `micro02` é o núcleo da rede, ele deve ser ligado primeiro para criar as portas de conexão.

1. **No CMD 1**, inicie o roteador e aguarde o boot:
   ```cmd
   micro02.bat
   ```
2. **Nos CMDs 2 e 3**, inicie as máquinas das pontas e aguarde:
   ```cmd
   micro01.bat
   ```
   ```cmd
   micro03.bat
   ```

### 3. Teste de Isolamento (No Micro01)
Vamos tentar acessar a outra rede e a internet a partir do `micro01`. Vá para o terminal do **micro01** e digite:

```bash
# Tentando pingar o Micro03
ping -c 4 192.168.1.10 

# Tentando pingar o Google
ping -c 4 8.8.8.8
```
🔴 **O que acontece?** Ambos os pings vão **falhar**. 
**Por quê?** Embora o `micro02` esteja fisicamente conectado às duas redes, por padrão, o Linux age apenas como um computador (host) comum. Ele recebe o pacote do `micro01`, vê que o destino final não é ele mesmo e descarta o pacote por segurança.

### 4. A Mágica do Roteamento (No Micro02)
Para que o `micro02` passe os pacotes de uma rede para a outra, precisamos habilitar o **IP Forwarding** (Encaminhamento de IP).

Vá para o terminal do **micro02** e digite:
```bash
echo 1 > /proc/sys/net/ipv4/ip_forward
```
*(Nota: Este comando avisa ao kernel do Linux que ele agora tem permissão para atuar como roteador).*

### 5. A Prova do Roteamento (No Micro01)
Volte para o terminal do **micro01** e repita os testes:

```bash
# Tentando pingar o Micro03 novamente
ping -c 4 192.168.1.10 
```
🟢 **Funciona!** O pacote vai até o `micro02`, que agora roteia o tráfego e entrega ao `micro03`.

```bash
# Tentando pingar o Google novamente
ping -c 4 8.8.8.8
```
🔴 **Ainda falha.** **Por quê?** O IP do `micro01` (`192.168.0.10`) é um **endereço privado**. A internet pública não sabe como devolver a resposta para esse endereço. Precisamos do NAT.

### 6. Configurando o NAT para acesso à Internet (No Micro02)
Para que a rede interna acesse a internet, o roteador precisa "mascarar" os pacotes internos com o seu próprio IP que tem acesso à rede externa. 

Vá para o terminal do **micro02** e crie a regra de NAT no `iptables` (assumindo que `eth0` é a interface conectada à internet):

```bash
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
```

### 7. Teste Final: O Efeito Inception (No Micro01)
Volte ao **micro01** e teste a internet:

```bash
ping -c 4 8.8.8.8
```
🟢 **Sucesso!** Agora você tem internet.

Para vermos como a internet está enxergando nossa máquina, vamos pedir para um site externo nos dizer qual é o nosso IP:
```bash
curl ifconfig.me
```

**O que significa o IP retornado?**
O IP exibido **não é** `192.168.0.10`. No nosso laboratório, temos uma cadeia de múltiplos NATs:
1. O `micro01` é mascarado pelo `micro02` (NAT interno do laboratório).
2. O `micro02` é mascarado pelo emulador QEMU / seu PC físico.
3. O seu PC físico é mascarado pelo Roteador da sua Operadora (que detém o IP público exibido pelo `curl`).

Obs: para chegar até a intenet temos vários nats 

micro01 - 192.168.0.10 -> qemu 192.168.2.15 -> computador - 192.168.x.y -> roteador etec -> ip publico 
todos trocam o ip com NAT
