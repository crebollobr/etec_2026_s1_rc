## 📅 A Camada de Aplicação e o Cliente-Servidor

Agora que sabemos como os pacotes chegam ao destino (graças ao IP e aos roteadores), vamos focar em como os **programas** conversam (Camada de Transporte e Aplicação). 

### 📖 Teoria (1 hora)


#### 1. Revendo as Camadas e o Cliente-Servidor
* Física/Enlace (Cabos), Rede (IP), Transporte (Portas/Conexão) e **Aplicação** (O Navegador, o Jogo, o E-mail).
* No modelo **Cliente-Servidor**, o Cliente (você) é sempre quem inicia a conversa. O Servidor fica apenas ouvindo, esperando alguém chamar.

#### 2. Transporte: TCP vs UDP e as Portas
* **TCP:** Estabelece uma conexão formal antes de falar (o famoso *3-Way Handshake*). É confiável.
* **UDP:** Envia os dados sem avisar e sem criar conexão. É rápido.
* **Portas e o "Bind":** Um IP identifica o computador, mas a **Porta** identifica o *programa* dentro dele. Para um servidor funcionar, o programa precisa fazer um *Bind* (se "amarrar") a uma porta e ficar escutando.
  * **Portas Baixas (0 a 1023):** Somente o usuário `root` (administrador) pode abrir. Ex: Porta 80 para sites. Isso evita que um usuário comum crie um servidor falso na máquina.
  * **Portas Altas (1024+):** Usadas pelos clientes. 

#### 3. A Mágica das Abas do Navegador
Se você abre 5 abas do YouTube, por que o vídeo de uma aba não aparece na outra?
* O destino é o mesmo: IP do YouTube, Porta 443.
* A origem muda! O seu sistema operacional escolhe uma **porta alta aleatória diferente** para cada aba (Ex: Aba 1 usa a porta origem 50001, Aba 2 usa a 50002). É assim que o computador sabe para qual aba entregar cada pedaço de vídeo que chega!

#### 4. Simulação Detalhada: O Pedido da Página do Google
1. **O Handshake (TCP):** O cliente (Porta 54321) diz `SYN` para o Servidor (Porta 80). O servidor diz `SYN-ACK`. O cliente confirma com `ACK`. Estão conectados!
2. **O Pedido (Aplicação):** O cliente envia um texto simples dizendo: `GET / HTTP/1.1` (Me dê a página inicial).
3. **A Resposta:** O servidor do Google responde com o código HTML da página.

### 💻 Laboratório Prático 02 (1 hora)

Vamos simular o navegador web de forma manual pelo terminal. Nós faremos o papel do navegador, nos conectaremos na porta 80 do Google e enviaremos o comando `GET`.

**Passo 1: Conversando diretamente com o Servidor Web**
Usaremos a ferramenta `nc` (Netcat) para criar uma conexão crua na porta 80 de um site que aceita requisições simples.

1. Conecte-se ao servidor na porta 80:
   ```bash
   nc www.google.com 80
   ```
2. *Atenção:* O terminal ficará piscando, esperando você digitar. Você acabou de fazer o 3-Way Handshake! Agora, digite o pedido `GET` exatamente como abaixo e pressione `Enter` **duas vezes**:
   ```http
  GET / HTTP/1.0
   
   ```
   *O que aconteceu?* A tela foi inundada de código HTML! Você acabou de simular manualmente o que o Google Chrome faz por baixo dos panos todos os dias.

**Passo 2: Vendo as Portas Altas e Baixas**
Vamos abrir uma conexão real com o `curl` e ver as portas em uso na nossa máquina.

1. Faça uma requisição web que demora alguns segundos (para dar tempo de olharmos as portas):
   ```bash
   curl [https://httpbin.org/delay/5](https://httpbin.org/delay/5) &
   ```
2. Rapidamente, liste suas conexões ativas:
   ```bash
   ss -npt | grep httpbin
   ```
   *Observe o resultado:* Você verá o seu IP local usando uma **porta alta** (ex: 48932) conectado ao IP do destino na **porta baixa** 443 (https). É a teoria acontecendo na prática!
