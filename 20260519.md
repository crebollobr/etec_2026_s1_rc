# Aula prática — Servidor SSH em container Docker

**Ambiente:** Killercoda Ubuntu Playground (https://killercoda.com/playgrounds/scenario/ubuntu)
**Pré-requisitos:** navegador com acesso liberado a `killercoda.com` via proxy.

---

## Princípios iniciais (slide de abertura)

- **Container ≠ VM.** Imagem Ubuntu base vem mínima. Qualquer comando além de `apt`, `ls`, `cat`, `sed`, `useradd` precisa ser instalado. Quanto menor a imagem, menor a superfície de ataque.
- **FTP não será configurado na prática.** Foi superado pelo SSH/SFTP — criptografia nativa, autenticação por chave, uma porta só. FTP sobrevive apenas em sistemas legados. Transferência de arquivos será demonstrada com `scp` e `sftp`.
- **Acesso externo na porta 2222 não funciona.** Killercoda só expõe HTTPS via proxy, não TCP cru. Cliente e servidor moram na mesma VM (dois terminais).

---

## Etapa 1 — Host Killercoda (terminal 1): subir container

```bash
docker rm -f lab 2>/dev/null
docker run -d --name lab -p 2222:22 ubuntu sleep infinity
docker ps
```

`docker ps` deve mostrar mapeamento `0.0.0.0:2222->22/tcp`.

Entrar no container:

```bash
docker exec -it lab bash
```

## Etapa 2 — Dentro do container: instalar dependências

```bash
apt update
apt install -y openssh-server iproute2 nano procps
```

| Pacote | Por quê |
|---|---|
| `openssh-server` | traz `sshd` + cliente (`ssh`, `scp`, `sftp`, `ssh-keygen`) |
| `iproute2` | comando `ss` para listar portas |
| `nano` | editor para `sshd_config` |
| `procps` | `ps` e `pkill` para gerenciar processo |

## Etapa 3 — Criar usuário e subir o sshd

```bash
mkdir -p /run/sshd

useradd -m -s /bin/bash aluno
echo "aluno:senha123" | chpasswd

/usr/sbin/sshd
ss -tlnp | grep :22
ps aux | grep sshd

exit
```

> **Observação:** rodamos `sshd` direto (sem `systemctl`) porque container não tem systemd como init. `sshd -d` é modo debug, atende uma única conexão e sai. Para uso normal, rode sem flag.

## Etapa 4 — Cliente: conectar (terminal 2 do Killercoda)

Abra uma segunda aba (botão `+` na barra do Killercoda). Você está no host, que já tem `ssh` instalado.

```bash
ssh -p 2222 aluno@localhost
# yes para aceitar a fingerprint do host
# senha: senha123
```

Saia com `exit`.

## Etapa 5 — Autenticação por chave (momento "aha")

No terminal 2:

```bash
ssh-keygen -t ed25519 -f ~/.ssh/aula -N ""
ssh-copy-id -i ~/.ssh/aula.pub -p 2222 aluno@localhost
ssh -i ~/.ssh/aula -p 2222 aluno@localhost
```

Login sem pedir senha → a chave pública foi gravada em `~/.ssh/authorized_keys` do usuário `aluno` dentro do container.

## Etapa 6 — Endurecer o servidor

Volta ao container:

```bash
docker exec -it lab bash

sed -i 's/#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sed -i 's/#\?PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config

sshd -t                  # valida sintaxe antes de reiniciar
pkill sshd
/usr/sbin/sshd

exit
```

## Etapa 7 — Validar o endurecimento

No terminal 2:

```bash
ssh -p 2222 aluno@localhost                       # deve FALHAR (Permission denied)
ssh -i ~/.ssh/aula -p 2222 aluno@localhost        # deve FUNCIONAR
```

## Etapa 8 — Transferência de arquivos (substituindo FTP)

No terminal 2:

```bash
echo "conteudo de teste" > arquivo.txt

# SCP — cópia simples
scp -i ~/.ssh/aula -P 2222 arquivo.txt aluno@localhost:/tmp/

# SFTP — sessão interativa estilo FTP, criptografada
sftp -i ~/.ssh/aula -P 2222 aluno@localhost
# dentro do sftp:
#   ls, pwd, cd /tmp, get arquivo.txt, put outro.txt, bye
```

## Etapa 9 — Limpeza

```bash
docker rm -f lab
```

## Checkpoints de validação

| Etapa | O que validar |
|---|---|
| 1 | `docker ps` mostra `0.0.0.0:2222->22/tcp` |
| 2 | `apt install` termina sem erro |
| 3 | `ss -tlnp` mostra `*:22 LISTEN`; `ps` mostra processo `sshd` |
| 4 | Prompt muda para `aluno@<hash>:~$` |
| 5 | `ssh` com chave não pede senha |
| 7 | Senha rejeitada, chave aceita |
| 8 | `arquivo.txt` aparece em `/tmp/` do container |

## Comparativo FTP × SSH/SFTP (slide de fechamento)

| Aspecto | FTP (1985, RFC 959) | SSH/SFTP (1995) |
|---|---|---|
| Criptografia | Nenhuma (texto claro) | Sempre |
| Portas | 21 controle + dados (hostil a NAT) | Apenas 22 |
| Autenticação | Senha em claro | Chave pública / MFA |
| Atravessar firewall | Complexo (modo ativo/passivo) | Simples |
| Status atual | Legado | Padrão da indústria |

**Por que FTP foi superado:** com a popularização do SSH e seu subsistema SFTP, manter um protocolo separado, inseguro e complicado tornou-se redundante. Hoje FTP só sobrevive em mainframes, EDI bancário antigo e alguns CDNs.

## Anexo — Troubleshooting comum

**"Package openssh-server has no installation candidate"**
Rode `apt update` antes do `apt install`.

**"Missing privilege separation directory: /run/sshd"**
Execute `mkdir -p /run/sshd`.

**"System has not been booted with systemd as init system"**
Container não tem systemd. Use `/usr/sbin/sshd` direto, não `systemctl start ssh`.

**"kex_exchange_identification: client sent invalid protocol identifier 'GET / HTTP/1.1'"**
Sondagem HTTP bateu na porta — provavelmente o port-checker do Killercoda. Ignore, o sshd está funcionando.

**"ssh: connect to host ...killercoda.com port 2222: Connection refused"**
Killercoda não expõe TCP externo. Teste apenas de dentro do playground (terminal 2 → `localhost:2222`).
